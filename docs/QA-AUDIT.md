## Planned security / IDOR audit (not yet done)

Before wider rollout, a dedicated security pass is needed:

1. **Fine-grained per-role RLS** — agent should only see their own leads (`created_by = auth.uid()`); TL sees team; Admin sees all. Currently all authenticated users see all data (baseline only).
2. **Admin endpoint auth guard** — `/api/users/create` and `/api/users/reset-password` should verify the calling user's role via their JWT before executing. No guard exists today.
3. **IDOR on notify endpoint** — `POST /notify` currently accepts any payload; a logged-in user could trigger email to arbitrary addresses. Should verify sender and recipient are real users.
4. **Approval flow double-execute** — two approvers acting simultaneously can both approve the same report. Add optimistic-concurrency guard (`.eq('report_approval','Pending')` before update).
5. **Confirm checkbox terminal-status guard** — "Mark confirmed by prospect" should be blocked if meeting status is Completed/Missed/Cancelled (QA audit item #9, High — listed above as still open).

### FIX FIRST — confirmed Critical / High issues

These are the confirmed critical and high-severity defects that should be addressed before any further work or deployment. Ordered roughly by blast radius.

1. **RLS is OFF on all 70 tables + anon has full DML** — anyone with the public anon key can read/write/delete the entire CRM. `lib/supabase.ts` / whole DB. _(Critical, confirmed)_
2. **Privilege escalation via `profiles.role`** — any authenticated user can self-promote to ADMIN with one PATCH. `DB: profiles` / role-gate logic. _(Critical, confirmed)_
3. **New-company inline creation inserts non-existent columns** — every new-company lead create/edit silently fails and drops the company name. `lib/leadsApi.ts:498-510`. _(Critical, confirmed)_
4. **Hardcoded `industry_id: 1` violates FK** — new-company creation fails for a second independent reason. `lib/leadsApi.ts:504`. _(Critical-pair, confirmed)_
5. **`lead_master.created_by` stored a NAME instead of user_id** — breaks ownership/RLS keying; live latent bug in the create path. `DB: lead_id=1` / `pages/LeadFormPage.tsx:249-252`. _(High, confirmed)_
6. **Rejected reports are locked** — the promised "edit and resubmit" flow is impossible; rejected reports are a dead end. `data/leadWorkspace.ts:693-696` + `ReportTab.tsx:806`. _(High, confirmed)_
7. **Saved 'No'/'Tentative' meeting decisions never re-hydrate** — prior decision is invisible on reload and silently lost on re-save. `ReportTab.tsx:114-151` + `leadWorkspace.ts:394-436`. _(High, confirmed)_
8. **Meetings module reads Company/Industry/City from the wrong source** — blank for ~78% of meetings (the leads module was already fixed; meetings wasn't). `data/meetings.ts:333-360`. _(High, confirmed)_
9. **'Mark confirmed by prospect' has no status guard** — overwrites terminal Completed/Missed status and regresses lead stage 8→5 on ~395 concluded meetings. `data/meetings.ts:797-815`. _(High, confirmed)_
10. **Wishlist Reassign Agent dropdown built from wrong source** — current agent silently disappears for ~10 of 11 assigned agents. `data/wishlist.ts:383-431` + `AssignModal.tsx`. _(High, confirmed)_
11. **Editing a user's role hard-DELETES all their other roles** — multi-role users (e.g. user 24 has 4 roles) lose all but one on Save. `data/admin.ts:224-243`. _(High, confirmed)_
12. **91% of notifications have `user_id = NULL`** — invisible to everyone; Notifications page looks empty despite a full table. `data/account.ts:51-57`. _(High, confirmed)_
13. **New notifications set `status='unread'`, rendered as the notification TITLE** — every new approval notification shows the literal word "unread" as its heading. `data/approvals.ts:89` + `NotificationsPage.tsx:23-25`. _(High, confirmed)_
14. **RLS-off makes mark-seen / profile-update unscoped** — cross-user writes possible. `data/account.ts:70-101`. _(High, confirmed)_
15. **`xlsx@0.18.5` has two unpatched HIGH CVEs** (Prototype Pollution + ReDoS) and is statically bundled into the main chunk. `package.json:21`. _(High, confirmed)_

> Note: items 5–15 are all High and all confirmed; address the three Criticals (1–4) first as they expose the entire database.
